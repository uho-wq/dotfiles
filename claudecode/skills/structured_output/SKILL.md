# Structured Output

出力が読みにくい根本原因は、長さよりも**構造の欠如**であることが多い。
Claude Codeは作業順・思いついた順に linear に書きがちで、その結果

結論が末尾に埋もれる
抽象度の違うもの（結論・根拠・場所・対策案）が同列に並ぶ
関連する点があちこちに散らばる

長さや冗長さは、この構造不全の**症状**にすぎない。先に構造を整えれば、読みやすさも簡潔さも後からついてくる。

## 目指す形：ピラミッド

```
        主メッセージ（結論を1つ）
        ├── 根拠グループA
        │     └── 詳細
        ├── 根拠グループB
        │     └── 詳細
        └── 根拠グループC
```

読み手が上だけ読めば要旨、下りれば根拠が分かる。これが「構造化された」状態。

---

## 原則

**頂点に主メッセージを1つ置き、冒頭に出す。** 出力全体が「結局何が言いたいのか」を一文で言えること。言えないなら、まだ構造が決まっていない。
**時系列ではなく結論で組む。** 「Xを見て、Yを見て、だからZ」ではなく「Zだ。根拠は(A)(B)」。作業ログは構造ではない。何をしたかではなく、何が分かったかで並べる。
**抽象度をそろえ、グループ化する。** 同じ階層には同じ粒度のものだけを置く。フラットに並んだ5項目が、実は2グループなら2つに畳む。
**見出し・箇条書きの先頭は「ラベル」でなく「メッセージ」にする。** 「## 変更点」ではなく「## レスポンス形式が変わる」。見出しだけ拾って筋が通るのが理想。
**横はMECE、縦はQ&Aでつなぐ。** 同じ階層の項目は重複なくモレなく、かつ同じ種類（全部「原因」か、全部「対策」か）。上の項目が呼ぶ「なぜ?／どうやって?」に、下の階層が答える形にする。

構造を決めた時点で、構造の外にあるノイズ（前置き・作業実況・「念のため」などの定型クッション）はピラミッドのどこにも入らないので、自然に落ちる。**簡潔さは独立した目標ではなく、構造化の副産物。**

## Before / After（長さではなく構造の違い）

### 例1：調査結果の報告

**Before（時系列・平地）:**
ログを見たところリクエスト自体は届いていました。それから接続まわりを調べ、プール設定が最大10だと分かりました。アクセスログでは深夜に集中していて、タイムアウトも出ています。設定は `config/db.ts` です。プールを増やすか、リークがないか確認するのがよさそうです。監視も入れるといいかもしれません。

→ 結論・根拠・場所・対策が全部同列で、最後まで読まないと要点が掴めない。

**After（ピラミッド）:**
**原因は接続プールの枯渇です。**
根拠:
- 深夜のアクセス集中時、同時接続が上限の10を超えて待機→タイムアウトしている
- リクエストは正常に到達しており、アプリ層ではなく接続層の問題
>
対策は、まずプールサイズの引き上げ。並行してリークの有無も確認すべき（リークが原因なら増やしても再発するため）。

→ 1行目で結論。根拠は「原因の根拠」だけでMECEにまとめ、対策は別の階層に分離。

### 例2：変更の説明

**Before:**
`auth.ts` を編集しました。`verifyToken` という関数を追加して、これはトークンを検証します。あと `routes.ts` も変えて、この関数を呼ぶようにしました。それから型も足しています。

**After:**
**保護ルートにJWT検証を追加しました。** 変更は2ファイル:
- `auth.ts`: 検証ロジック（`verifyToken`）を新規追加
- `routes.ts`: 保護ルートで `verifyToken` を呼ぶように変更

## 構造化のセルフチェック

[ ] 主メッセージを一文で言えるか（言えないなら構造が未定）
[ ] 1行目が結論になっているか
[ ] 見出し／要点を上から拾うだけで筋が通るか
[ ] 同じ階層の項目は、同じ粒度・同じ種類か（原因と対策が混ざっていないか）
[ ] 時系列・作業順で並べていないか
[ ] フラットな箇条書きが、実はグループ化できないか

## 成果物をHTMLで出力する

構造化した結果は、プレーンなテキストではなく**自己完結した単一HTMLファイル**として出力する。ピラミッド構造を視覚的な階層に対応させることで、「上から読めば要旨、下りれば根拠」を見た目でも実現する。

参照テンプレート: `assets/template.html`。このCSS設計（スパインによる階層表現・結論の強調・`<details>` による詳細の段階的開示）をベースにし、内容に合わせて中身を差し替える。

### ピラミッド → HTML の対応

| 論理構造 | HTML上の表現 |
| --- | --- |
| 主メッセージ（頂点・結論1つ） | 最上部の見出し。アクセント背景で強調し、これだけ読めば要旨が分かる |
| 役割ラベル（結論／根拠／対策） | 各ブロックの eyebrow。論理的役割を encode するもので、装飾ではない |
| 根拠グループ（MECE・並列） | メッセージ見出し付きのセクション。スパイン上のノードで並列性を示す |
| 詳細（ログ・コード・補足） | `<details>` で折りたたむ。読み手が必要なときだけ開く（段階的開示） |
| ファイルパス・コード | `<code>` / `<pre>`（等幅フォント） |

### 守ること

**単一ファイルで自己完結。** CSSは `<style>` にインライン。外部依存・CDN・画像参照を作らない。
**見出しだけ拾って筋が通る**よう、見出しはラベルでなくメッセージにする（原則4と同じ）。
**モバイル対応・キーボードフォーカス可視・`prefers-reduced-motion` 尊重**を最低ラインとして満たす。
出力先は作業ディレクトリ配下に保存し、パスをユーザーに伝える。

### HTMLにする/しないの線引き

HTMLにするのは、ユーザーが**保存・共有・後で読み返す**ような実体のある成果物のとき（調査レポート、分析、変更サマリ、複数論点の説明）。
「ファイルがあったので新規作成しました」程度の一言の確認や、その場で完結する短い応答までHTML化しない。その場合は本文の構造化原則（結論先・グループ化）をテキストのまま適用する。

---

## やりすぎ防止

**構造化＝見出しや箇条書きを増やすことではない。** 最小の構造は「一文の中で結論を先、理由を後」。論点が複数あるときだけグループ化・箇条書きを使う。短い答えは一文で構造化できる。
一言で済む応答にまでピラミッドの枠を被せない。同じく、短い確認や雑談までHTMLファイルにしない。
構造を優先するあまり、**実際の答え・破壊的操作の事前確認・想定と違う前提で進めた事実**を省略しない。これらは必ず残す。
