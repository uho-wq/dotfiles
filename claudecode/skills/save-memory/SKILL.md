# Session Insights → Auto Memory 保存ガイド

セッション中に得た新しい知見・パターン・決定事項をauto-memoryに保存する。

## メモリの構造

memoryディレクトリ (`~/.claude/projects/<project-path>/memory/`) は以下の構成:

- **MEMORY.md** -- インデックスファイル。トピック一覧テーブル + Quick Notes。毎セッション自動読み込み（200行上限）
- **`<topic>.md`** -- トピック別の詳細ファイル。必要時にのみ読み込まれる

## ワークフロー

### Step 1: 会話の振り返り

このセッションの会話全体を振り返り、以下のカテゴリで新しい知見を特定する:

| カテゴリ | 例 |
|---------|-----|
| **安定したパターン・規約** | コーディング規約、命名規則、アーキテクチャパターン |
| **重要な設計決定** | 技術選定の理由、トレードオフの判断 |
| **デバッグ知見** | 再発しそうな問題の原因と解決策 |
| **ユーザーの好み** | ワークフロー、ツール、コミュニケーションスタイルの好み |
| **重要なファイルパス** | よく参照するファイル、設定ファイルの場所 |

### Step 2: 既存メモリの確認

1. MEMORY.md を読み、トピック一覧テーブルを確認する
2. 新しい知見が既存トピックに該当するか判定する
3. 該当するトピックファイルがあれば、そのファイルも読んで重複を確認する

**重複チェック**: 既に記録されている知見は追加しない。

### Step 3: フィルタリング

以下に該当するものは **保存しない**:

- セッション固有の一時的な情報（進行中のタスク、作業状態）
- 1回限りの操作で再利用価値がない情報
- 不完全・未検証の推測
- 既存のCLAUDE.mdに記載済みの内容と重複する情報
- 一般的なプログラミング知識（プロジェクト固有でないもの）

### Step 4: メモリへの書き込み

知見がある場合のみ、以下の手順で保存する。

#### 4-1: トピック設計の検討

今回の知見をどう整理するか、自分自身で検討する:

- 既存のトピックファイルに追記すべきか、新しいトピックファイルを作るべきか、Quick Notesに仮置きすべきかを判断する
- **判断基準は「将来の自分（別セッションのClaude）が探しやすいか」**
- 既存トピックと重なりつつも視点が違う場合は、既存トピックの拡張か新規トピックかを検討する
- Quick Notesが肥大化していたら、関連する項目をまとめてトピックファイルに昇格させることも検討する

#### 4-2: 書き込み実行

- **トピックファイル**: kebab-caseの英語名で `memory/` ディレクトリに配置（例: `hooks.md`, `zsh-config.md`）
- **MEMORY.md**: トピックファイルを作成/更新したらテーブルを必ず同期する
- **Quick Notes**: まだトピックにまとめるほどでない断片的知見の一時置き場

#### MEMORY.md のフォーマット

```markdown
# <Project Name> Project Memory

## Topics

| Topic | File | Summary |
|-------|------|---------|
| トピック名 | [ファイル名](ファイル名) | 1行の簡潔なサマリー |

## Quick Notes
- 断片的な知見をここに
```

### Step 5: 完了報告

保存した知見の要約をユーザーに報告する:
- どのトピックファイルに何を追記/作成したか
- Quick Notesに追加した場合はその内容
- 知見がなかった場合は「新しい知見はありませんでした」と報告する

## 注意事項

- auto-memoryはプロジェクトごとに独立している
- MEMORY.mdは毎セッション開始時に自動読み込みされる（200行上限、超過分は切り捨て）
- 別トピックファイルは自動読み込みされない。MEMORY.mdのテーブルにサマリーがあることが重要
- 誤った情報を保存した場合、次セッションの判断に悪影響を及ぼすため、正確性を重視する
- MEMORY.mdのテーブルとトピックファイルの整合性を常に保つ
