### 構造化の原則（structured_output 準拠 / Markdown 版）
読みにくさの根本原因は長さより**構造の欠如**。先に構造を整えれば簡潔さは後からついてくる。

- **冒頭に結論を1つ。** PR全体を一文で言えること（＝ `## TL;DR`）。言えないなら変更の主旨がまだ整理できていない。
- **時系列でなく結論で並べる。** 「まずAして、次にBして…」という作業ログにしない。「Xを実現した。変更は(1)(2)」の順で書く。
- **各セクション内も結論先・MECE。** 同じ見出し下の箇条書きは、重複なくモレなく、同じ粒度・同じ種類でそろえる。フラットな5項目が実は2グループなら畳む。
- **「やったこと」は差分のなぞり書きにしない。** 行単位の説明（「関数Xを追加し、Yを呼ぶよう変更」）はdiffを読めば分かる。レビュワーがdiffを読む前に掴むべき"変更の意味"を、ファイル／論点単位でまとめる。
- **中身のないセクションは埋めず削る。** 「特になし」と書くより見出しごと省略する（トレードオフ・フロー図など）。全セクションを等量で埋めようとしない。

### AIっぽさを消す（禁止する癖・表現）
- クッション句・ヘッジを書かない:「念のため」「〜かもしれません」「〜だと思われます」「基本的には」。判断は言い切る。
- 自明な前置き・締めを書かない:「本PRでは〜を行います」「以上が変更内容です」。
- 過剰な装飾をしない: 絵文字の多用、マーケ調の形容（「シームレスに」「堅牢な」）、強調の乱用。
- 箇条書きを水増ししない: 1項目で済むものを分割しない。本当に並列な論点のときだけ箇条書きにする。

### コンテキスト収集のルール
PRの説明文には「コード（差分そのもの）」「設計の意図」「トレードオフの判断」「変更の影響範囲」の4つの観点を含めること。

以下の順序で情報を収集する:
1. コードの差分・コミット履歴から読み取れる情報をまず整理
2. claude-mem MCPおよびClaude Codeのauto memoryを検索し、意思決定プロセスを探す
3. **1・2で設計意図やトレードオフの判断が読み取れない場合は、必ずユーザーに質問する。推測で埋めてはならない**

### 提出前のセルフチェック
- [ ] `## TL;DR` だけ読んでPRの主旨が分かるか
- [ ] 見出しを上から拾うだけで筋が通るか
- [ ] 作業順・時系列で書いていないか
- [ ] diffを読めば分かることを冗長になぞっていないか
- [ ] 中身の薄いセクションを無理に埋めていないか
- [ ] クッション句・ヘッジ・自明な前置きが残っていないか

Co-Authored-Byを含めるのは禁止します。
